## Task 1 — F-Find (search → select + jump)

**Find flow:**
1. Press `F` (window `keydown`) → the overlay opens. The handler ignores the key
   when the event target is an `INPUT`/`TEXTAREA`/`contenteditable`, when the box
   is already open (`!findOpen`), or when any modifier (Ctrl/Meta/Alt) is held —
   so typing `f` inside the find box types a letter, and Ctrl+F stays the
   browser's own find.
2. Type part of a tech name → case-insensitive `includes` match against **all**
   techs (built from `snapshot.techs`, incl. filter-hidden ones), top ~12 rows
   (icon + name + `Category · Tier`, area-colored left border).
3. Navigate: `↑`/`↓` move the active row (wrapping), `Enter` picks it, `Esc`
   closes; rows are also hover-highlight + click, and a backdrop click closes.
4. `onPick(key)`: un-hides the tech's `category[0]` if it's filtered out
   (`setActive`), `setSelectedKey(key)` (gold selection + edge highlight +
   ancestry panel reuse), looks the node up in `state.layout.nodes` and
   `fitToNodes([node])` to pan/zoom-jump to it, then closes the overlay.

**New file:** `app/src/components/FindOverlay.tsx` (presentational; exports
`FindEntry`). **Touched:** `TechTree.tsx` (findOpen state, keydown effect,
`findEntries` memo, `onPick`, render), `app.css` (`.find-overlay` / `.find-box`).

## Task 3 — Weight-modifier detail ("what boosts find-chance")

**New pure helper** `app/src/lib/graph/weight.ts` —
`describeWeightModifiers(raw: unknown, techByKey: Map<string, Tech>): string[]`:

- Top-level `factor` (if present and ≠ 1) → `"Base ×{factor}"`.
- `modifier` is normalized (single object OR array — duplicates preserved) →
  each entry → `"×{factor} if {conditions}"`.
- Condition formatting:
  - `has_technology: "tech_x"` → `researched {resolved name}` via `techByKey`
    (falls back to a prettified key).
  - `{ GREATER_THAN: 2 }` / `{ LESS_THAN: 2 }` etc. → `> 2` / `< 2`
    (`GREATER_THAN→>`, `LESS_THAN→<`, `≥`, `≤`, `=`, `≠`).
  - boolean flag → `with {flag}` / `without {flag}`.
  - plain scalar → `{humanized key} {value}`.
  - multiple conditions joined with `" and "`.
- **Defensive:** non-object input, deep logic blocks (OR/NOR/NOT/AND, nested
  federation/TYPE objects) and unknown shapes are skipped, never thrown on;
  unresolved `@variable` factors are preserved literally.

**Formatting examples (real tech.json v4.5.0):**

| Tech | Output lines |
| ---- | ------------ |
| `tech_afterburners_1` (w 95) | `Base ×1.25` · `×2 if researched Ion Thrusters` · `×2 if researched Plasma Thrusters` |
| `tech_adaptive_bureaucracy` (w 95) | `×0.5 if owned planets < 2` · `×1.5 if owned planets > 2` · `×1.5 if owned planets > 3` · `×1.5 if owned planets > 4` |
| `tech_adaptive_combat_algorithms` (w 75) | `Base ×1.5` |
| `tech_advanced_industrial_storm_protection` (w 85) | `×0.5 if cosmic storms encountered < 4` |
| `tech_alien_cloning` (w 85) | `×@federation_perk_factor if with federation` |
| `tech_administrative_ai` (w 100) | (none — only a nested TYPE object, skipped) |

Across the dataset, **394 / 678** techs produce at least one readable line.

`TechTooltip` gained a **"Research Weight"** section (base `Base weight {weight}`
