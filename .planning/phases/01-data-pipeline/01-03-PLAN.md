<objective>
Build the localisation layer that resolves every tech's English name and description from the game's `.yml` files, so the snapshot ships real display text (DATA-03) rather than raw tech keys. This is a self-contained slice: it consumes the tech keys produced by the extractor and produces a name/description resolver that assemble.ts (Plan 05) applies to every tech.

Purpose: The reference tool's core failure was stale/wrong data; correct localisation is half of "accurate data presented clearly." RESEARCH.md proved this is trickier than it looks — tech localisation is scattered across 16+ files (only 48% lives in technology_l_english.yml) and uses two coexisting line syntaxes. Getting this wrong silently produces ~215 missing names.

Output: `scanAllLocalisation` (builds one global key->string map from every english .yml) and `resolveTechText` (resolves a tech key to its name and, when present, description).
</objective>

From pipeline/src/schema/tech-snapshot.ts:
- `Tech.name` (string) and `Tech.description` (nullable string) are the fields this plan's resolver feeds.

Verified localisation facts (RESEARCH.md Pitfalls 2 & 3, Code Example "Verified localisation line parser"):
- Tech localisation is scattered across 16+ .yml files. Only 328 of 678 tech keys live in technology_l_english.yml; the rest are in DLC-named files (megacorp_l_english.yml, ancient_relics_l_english.yml, first_contact_dlc_tech_l_english.yml, biogenesis_bioships_l_english.yml, etc). SCAN EVERY .yml — do NOT hardcode a file subset (a hardcoded list silently misses ~215 keys and breaks on future DLC).
- Two line syntaxes coexist: `key:0 "value"` (with numeric revision index) AND `key: "value"` (no index at all, e.g. tech_fe_lab_2 in fallen_empire_l_english.yml). The verified regex makes the index optional: `/^\s*([a-zA-Z0-9_.]+):(\d*)\s*"((?:[^"\\]|\\.)*)"/`. Requiring the digit falsely reports 215 keys missing.
- Files begin with a BOM and an `l_english:` header line — strip the BOM (charCodeAt(0)===0xfeff) and skip the header (it does not match the value regex anyway). Read files as utf8.
- A tech's name key is the tech key itself (e.g. `tech_space_exploration`). Descriptions typically live under `<tech_key>_desc` — resolve description from that key when present (nullable if absent; missing description is a cosmetic gap, warn-not-fail per D-16).
