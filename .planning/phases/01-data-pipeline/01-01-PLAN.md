- Read the file as a Buffer (readFileSync with no encoding), decode to latin1 for the wrap, and pass `parser.parseText(wrapped, 'windows1252')`.
- Verified to parse all 33 tech files + category + tier + all 22 scripted_variables files with zero errors.

Verified data shapes observed in the real corpus (ground truth for the schema):
- A tech block: `tech_space_exploration = { cost = 0  area = physics  start_tech = yes  category = { computing }  tier = 0 ... }`
- `area` is a bare string (physics|society|engineering). `tier` is a bare number. `category = { computing }` is a block/array (normalize to array).
- `cost` and `weight` are usually a bare number OR an `@variable` reference string (e.g. `weight = @tier1weight3`). Open Question 1 in RESEARCH.md: the block-form `cost = { factor = ... }` is documented-but-unobserved — the schema/extractor must accept both shapes defensively.
- `prerequisites = { "tech_basic_science_lab_1" }` — array of quoted tech keys.
- A tech's OWN unlock content (D-05 component a) lives in its definition as: `modifier = { <stat> = <n> }` top-level block(s) (stat bonuses granted), `feature_flags = { <flag> ... }` (named capability unlocks, e.g. `tech_automated_exploration`), `prereqfor_desc = { <kind> = { title = "<LOC_KEY>" desc = "<LOC_KEY>" } }` (structured unlock descriptions whose title/desc are localisation keys), and an optional `gateway = <string>`.
