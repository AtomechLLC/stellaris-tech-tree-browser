<objective>
Widen the walking skeleton's single-file, single-tech extraction into the full-fidelity tech data layer: parse all 33 tech files into the complete set of real `tech_*` definitions, populate every field (tier, area, category, resolved cost, resolved base weight, structurally-preserved weight_modifier, prerequisites, flags) INCLUDING each tech's own raw unlock content (the modifier/feature_flags/prereqfor_desc/gateway data that becomes the `unlocks.grants` text in Plan 05), classify each tech's gating DLC from authoritative sources, and build+validate the prerequisite DAG with computed reverse edges (`unlocks.leadsTo`). Purpose: This is the correctness core of the phase — the reference tool died from silently-wrong data, so this plan validates against the FULL 33-file corpus (D-18), fails loudly on structural errors (D-16), and preserves raw data faithfully for v2 (D-06). The prerequisite DAG is the structural backbone Phase 2 renders. Per D-05, `unlocks` has TWO components: (a) what the tech ITSELF grants (extracted here as raw content, localised in Plan 05) and (b) the reverse "leads to" edges (computed here). This plan produces the raw material for BOTH, so `unlocks` is never reduced to just reverse edges.

Output: `extractAllTechs` (now also emitting each tech's raw unlock content), `classifyDlc`, `loadDlcRegistry`, and `buildAndValidateGraph` — the modules assemble.ts (Plan 05) composes into the full snapshot.

Verified corpus facts (RESEARCH.md — all counted directly against real 4.5.0 files):
- 33 tech files in common/technology/*.txt (plus category/ and tier/ subdirs which are NOT tech files — skip subdir contents when enumerating tech files).
- 678 top-level `tech_*` definitions total. `000_documentation.txt` parses to 0 tech keys (a doc stub) — filter it out. Only keys matching `tech_*` are techs; skip non-tech top-level keys.
- Flags in the raw data: `is_rare = yes`, `is_dangerous = yes`, `start_tech = yes` (isStarting), and repeatable techs use `levels = -1` (or a positive levels value in *_repeatable.txt files) — isRepeatable is true when a `levels` field is present.
- `cost`/`weight` are usually bare numbers or `@variable` strings; block-form `cost = { factor... }` is documented-but-unobserved (Open Question 1 / A3) — handle both: if cost/weight is an object, store it in costRaw/weightModifierRaw and derive the numeric field defensively (use factor if present, else 0) rather than throwing.
- `weight_modifier` blocks: preserve raw; duplicate `modifier` children auto-array via jomini (D-06) — do NOT collapse.
- `technology_swap`: 207 blocks across 18 files; singular-or-array (Pitfall 5) — normalizeToArray before use.

Verified tech-own-unlock-content facts (D-05 component a — what the tech ITSELF grants; RESEARCH.md interfaces + verified in the real corpus):
- `feature_flags = { <flag> <flag> ... }` — a block of bare capability-unlock tokens (e.g. `tech_space_exploration` has `feature_flags = { tech_automated_exploration unlocks_auto_research }`). Capture the flag tokens.
- `prereqfor_desc = { <kind> = { title = "<LOC_KEY>" desc = "<LOC_KEY>" } }` — structured unlock descriptions whose `title`/`desc` values are LOCALISATION KEYS (e.g. `title = "TECH_UNLOCK_SCIENCE_SHIP_CONSTRUCTION_TITLE"`, which resolves in main_2_l_english.yml). `<kind>` can repeat (ship/building/component/etc) — normalizeToArray. Capture the title/desc loc-keys verbatim (do NOT resolve here — Plan 05 has the loc map).
- Top-level `modifier = { <stat> = <n> ... }` block(s) directly on the tech (NOT the ones nested inside weight_modifier/ai_weight — those are weight tuning, not grants). E.g. `tech_prescient_data_modeling` has `modifier = { planet_jobs_trade_produces_mult = 0.025 }`; `tech_eco_simulation` has both a top-level `modifier` and a `gateway`. A tech may have multiple top-level modifiers — normalizeToArray. Capture the raw stat key/value pairs (the stat keys are themselves localisation-key-like tokens Plan 05 can join, or ship verbatim if no loc entry).
- Optional `gateway = <string>` (e.g. `gateway = zone`) — capture verbatim.
- Distinguish top-level `modifier` (a grant) from `weight_modifier.modifier` / `ai_weight` (weight tuning, already handled by weightModifierRaw) — only the top-level `modifier` key on the tech object counts as a grant.

Verified DLC facts (RESEARCH.md Pitfall 4 + Don't Hand-Roll):
- `dlc/dlc0XX_*/dlc0XX.dlc` files (30 folders) are Clausewitz format with a `name = "..."` field (e.g. `name = "Ancient Relics Story Pack"`) that is the EXACT string used by in-script `host_has_dlc = "..."` triggers. Parse them with parseClausewitzFile.
- Primary DLC tag: filename convention (`00_ancient_relics_tech.txt` -> Ancient Relics), cross-checked to the .dlc registry names.
- Secondary/override: scan each tech's `potential` block for `host_has_dlc = "..."`. Only 11 occurrences across 3 files; exactly 2 techs are genuinely cross-DLC (`00_eng_tech.txt` -> Apocalypse and Federations, `00_strategic_resources_tech.txt` -> Distant Stars Story Pack). host_has_dlc, when present, OVERRIDES the filename tag.
- Base-game techs (no DLC filename match, no host_has_dlc) get dlc = null.

Verified graph facts:
- `prerequisites = { "tech_x" "tech_y" }` — array of quoted keys (use normalizeToArray). A dangling prerequisite (references a key not in the extracted set) is a structural error -> fail loud (D-16). Reverse edges ("leads to") = for each tech, the set of techs that list it as a prerequisite (D-05 `unlocks.leadsTo`). The graph must be acyclic — a cycle is a structural error.
