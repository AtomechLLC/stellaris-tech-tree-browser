<data_pipeline_reference>
From RESEARCH.md — the locked layout patterns for this plan:
- Pattern 3 "elkjs in a Vite/browser context": `import ELK from "elkjs/lib/elk.bundled.js"; const elk = new ELK();` — ZERO config, no workerUrl. Its in-process fake worker sidesteps the Vite worker-bundling bug class (Pitfall 4). Do NOT wire a real Web Worker.
- Pattern 2 "ELK tier-partitioning + post-layout area-band remap": root layoutOptions `elk.algorithm=layered`, `elk.direction=RIGHT`, `elk.partitioning.activate="true"`; per-node `elk.partitioning.partition = String(tech.tier)`. GOTCHA: per-node partition is silently ignored unless the ROOT sets `elk.partitioning.activate:"true"`. Spacing: start `elk.layered.spacing.nodeNodeBetweenLayers ~80`, `elk.spacing.nodeNode ~40` — TUNE against the real 678-node render (Claude's Discretion / Pitfall 5: generous spacing avoids border/image overlap artifacts).
- Pitfall 3 + Open Q2 "area band Y-remap": ELK has NO native swim-lane mechanism. After elk.layout() resolves: group result.children by tech.area, sort each group by ELK's own computed y (stable — preserves crossing-minimization), then offset each area into a fixed vertical band (physics band, society band, engineering band). Area order and band height are Claude's Discretion — tune against the real render. Implement the SORT-STABLE remap first (Open Q2 recommendation); only add a secondary sort if it looks poor.
- Anti-pattern (RESEARCH): do NOT import graphology-layout / forceAtlas2 — it would overwrite ELK's x/y. Do NOT rely on ELK's default edge-inferred layering (that IS the Treant.js bug D-06 fixes).
- Code Examples "buildGraph": `graph.addEdge(prereqKey, tech.key)` for every prerequisites[] entry; if `!graph.hasNode(prereqKey)` console.error (contract violation) rather than silently skip.
</data_pipeline_reference>
