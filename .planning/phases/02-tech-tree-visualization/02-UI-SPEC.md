# Phase 2 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none (deliberate — see Design System Rationale below) |
| Preset | not applicable |
| Component library | none — hand-built minimal chrome (header, tier-axis, legend, loading/error overlays); the graph canvas itself is Sigma/WebGL, not DOM components |
| Icon library | none — this phase uses game-data tech icons (WebP, from `pipeline/data/v{version}/icons/`) exclusively; no UI icon library needed for chrome (loading/error states use text, not icons) |
| Font | System UI stack (see Typography) |

### Design System Rationale

No `components.json` / shadcn detected in the repo (confirmed via `ls`). This is a greenfield React 19 + Vite 8 app. Ordinarily the shadcn gate would fire here. It is **deliberately skipped** for this phase:

- CONTEXT.md D-11/D-12 already lock a hand-rolled theming architecture: a single `tokens.css` of raw CSS custom properties (`--color-*`, `--space-*`, `--font-*`, `--area-*`), light theme as `:root` default, with Sigma's WebGL layer bridged via `getComputedStyle` reads at init (not Tailwind, not a component-library theme provider).
- shadcn's model assumes Tailwind utility classes and its own primitive components — introducing it now would mean either (a) re-litigating D-11/D-12 (CONTEXT.md instructs downstream agents not to re-open locked decisions) or (b) running two parallel theming systems, which directly undermines UIFX-01's "single token source" contract and the "dark mode must be a pure token swap" requirement.
- Phase 2's chrome surface is minimal by design (full-bleed canvas + header + tier-axis + legend + two overlay states) — there is no forms/CRUD surface where shadcn's primitives (Dialog, Select, Table, etc.) would pay for themselves. Phase 3 (search/filter/detail panel) is a more natural point to revisit whether a component primitive layer is worth adding, without contradicting this phase's locked contract.

Registry safety gate: not applicable (no shadcn, no registries).

## Spacing Scale

Declared values (multiples of 4, 8-point scale):

| Token | Value | Usage |
|-------|-------|-------|
| `--space-xs` | 4px | Icon-to-label gap inside node label chip, tier-axis tick spacing |
| `--space-sm` | 8px | Header internal padding (compact), legend swatch gaps |
| `--space-md` | 16px | Default chrome padding (header, legend container, overlay content) |
| `--space-lg` | 24px | Section padding (legend block, error/empty overlay card) |
| `--space-xl` | 32px | Layout gaps (space between header and canvas, overlay card margins) |
| `--space-2xl` | 48px | Major section breaks (not used at this phase's chrome density, declared for consistency) |
| `--space-3xl` | 64px | Page-level spacing (not used at this phase's chrome density, declared for consistency) |

Exceptions: **Sigma canvas coordinates are NOT CSS pixels** and do not follow this scale — node size, edge width, and inter-node spacing live in Sigma/graph coordinate space and are specified separately under Node Visual Spec and Edge Visual Spec below. The 8-point scale governs DOM/CSS chrome only (header, tier-axis labels, legend, loading/error overlays).

## Typography

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 16px | 400 (regular) | 1.5 |
| Label (node label, tier-axis label, legend text) | 14px | 400 (regular) | 1.2 |
| Heading (header title, overlay heading) | 20px | 600 (semibold) | 1.2 |
| Display (not used this phase — declared for Phase 3 continuity, e.g. a future large empty-state numeral) | 28px | 600 (semibold) | 1.2 |

**Font stack:** `--font-sans: -apple-system, BlinkMacSystemFont, "Segoe UI", Roboto, Helvetica, Arial, sans-serif` (system UI stack — zero web-font load cost, keeps first-paint fast, appropriate for a data-density tool where legibility matters more than brand typography).

**Node label rendering is NOT DOM/CSS** — Sigma renders labels on canvas using its own label renderer (font settings passed via Sigma's `defaultDrawNodeLabel` / renderer settings, sourced from the same `--font-sans` family and a Sigma-space font size tuned per Node Visual Spec below, not the 14px CSS value directly, since Sigma label sizing is in graph pixel units scaled by camera zoom).

Weights used: exactly 2 (400 regular, 600 semibold) — no light/medium/bold variants introduced.

## Color

Light theme only (`:root` default). All values below are CSS custom properties in `app/src/styles/tokens.css`.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `--color-bg: #F7F8FA` | App background, Sigma canvas background (`renderer.setSetting('renderLabels', ...)`-adjacent canvas clear color) |
| Secondary (30%) | `--color-surface: #FFFFFF` | Header bar, legend panel, loading/error overlay cards, tier-axis header strip |
| Accent (10%) | `--color-accent: #2563EB` | Reserved for: focus rings on interactive chrome (future search input in Phase 3), the "Retry" action in the error state, active/hover state of the (future) legend toggle. NOT used for node/edge default colors — those use the three area colors below, which are semantically separate from "accent." |
| Destructive | `--color-danger: #B91C1C` | Reserved for: error-state text/icon color and border in the data-fetch-failure overlay only. Not used elsewhere in Phase 2 (no destructive user actions exist in this phase). |

### Area colors (semantic categorical set — distinct from the 60/30/10 UI palette)

Per D-09, area is encoded by node border/ring color. These three colors must be mutually distinguishable under common color-vision deficiencies (deuteranopia, protanopia, tritanopia) since column/tier position is the *only* other structural cue and area needs to be independently legible. Using the Okabe-Ito colorblind-safe categorical palette (the same set recommended by Nature/Wong for scientific figures — verified via davidmathlogic.com/colorblind CVD simulation against blue/orange/vermillion, deuteranopia and protanopia safe, avoids the red/green pairing that fails ~8% of male users):

| Token | Value | Area | Notes |
|-------|-------|------|-------|
| `--area-physics` | `#0072B2` (blue) | Physics | |
| `--area-society` | `#E69F00` (orange) | Society | |
| `--area-engineering` | `#D55E00` (vermillion/burnt orange-red) | Engineering | Deliberately vermillion, not green — green paired with blue+orange is the classic deuteranopia failure mode; vermillion keeps all three legs distinguishable under simulated CVD. |

Ring rendering: each node's border/ring uses its area color at full opacity, ~2-3px stroke width in graph coordinate space (tune during implementation against the real 678-node render — the value must stay visible at default zoom without visually dominating the icon). Node fill/interior remains the icon's own WebP artwork (or placeholder) — the ring is a stroke, not a fill, so it never fights with icon color content.

**Do not rely on hue alone even with this safe palette**: area is reinforced structurally by swim-lane vertical position (D-06 — physics/society/engineering as three stacked horizontal bands), so a user who cannot distinguish the ring hue can still read area from row position. This is the required redundant-encoding pattern per the phase's accessibility note.

Accent reserved for: focus rings on future interactive chrome, error-state retry action. Never used for area/node/edge default styling — keeps "accent" (UI chrome semantic) and "area color" (data semantic) conceptually and visually separate so they don't compete for the same visual weight.

## App Shell Layout

Full-bleed canvas is the dominant surface. Chrome is minimal, non-modal, and never obstructs the graph:

| Region | Spec |
|--------|------|
| Header bar | Fixed top strip, 48px height, `--color-surface` background, `--space-md` horizontal padding, bottom border `1px solid #E5E7EB`. Contains: app title ("Stellaris Tech Tree") at Heading style, left-aligned. No search/filter controls here — Phase 3 scope. |
| Tier-axis header | Persistent horizontal strip directly below the header bar, 32px height, `--color-surface` background, bottom border `1px solid #E5E7EB`. Shows tier labels ("Tier 0", "Tier 1", ... "Tier 5") at Label style, horizontally positioned to align with each tier column's x-position in the Sigma viewport. Must stay pinned/aligned as the user pans horizontally (re-project tier-header x-offsets from Sigma camera state on pan/zoom events) — this is what makes TREE-04's "shows tier at a glance" hold without clicking, per D-09. |
| Canvas | Fills all remaining viewport space below header + tier-axis. `--color-bg` background. Sigma WebGL container, 100% width/height, no padding. |
| Legend | Small fixed-position overlay panel, bottom-left corner of the canvas, `--space-md` inset from the viewport edge, `--color-surface` background, `--space-sm` internal padding, subtle shadow (`0 1px 3px rgba(0,0,0,0.12)`) to lift it off the canvas. Shows the three area colors as a swatch + label list (e.g., a small circle in `--area-physics` + "Physics" text at Label style). Static for Phase 2 — not interactive/toggleable (that refinement is Phase 3-or-later discretion, not required now). |
| Loading state | See below. |
| Error state | See below. |

No sidebar, no modal, no bottom bar in Phase 2 — those are Phase 3 concerns (filters panel, detail panel).

## Node Visual Spec (TREE-04, D-09)

| Property | Spec |
|----------|------|
| Shape | Circular/square image-node via Sigma's built-in image node program (`@sigma/node-image` or core `NodeImageProgram`) |
| Size | Base radius 12px in graph coordinate space at zoom level 1 (tune against real render; must stay legible — icon recognizable — at the zoom level showing the full 678-node tree, and not overlap sibling nodes within the same tier column at ELK's default spacing) |
| Icon | The tech's WebP icon (`pipeline/data/v{version}/icons/{icon filename}.webp`), rendered filling the node's circular/square bounds. Object-fit equivalent: cover/contain per Sigma's image-node cropping behavior — use `contain` semantics so icon art is never cropped (Stellaris tech icons are meaningful pictograms, not photography). |
| Missing icon | Tech records with `icon: null` per SCHEMA.md never occur in practice (D-13's placeholder guarantees every tech has a shipped file) — but defensively, if `icon` resolves to a broken/unresolvable URL at runtime, render the same shipped placeholder WebP the pipeline already emits for that case. No separate "broken image" UI glyph needed — the placeholder already covers this. |
| Area ring | 2-3px stroke in the tech's area color (`--area-physics` / `--area-society` / `--area-engineering`), drawn as the node border. Full opacity. |
| Label | Localized `name` (from `tech.json`), rendered via Sigma's label renderer directly below or beside the node (Sigma default label placement), Label-style sizing translated to Sigma's font-size setting, `--font-sans` family, `#111827`-equivalent dark neutral text color (near-black, not pure black, for reduced glare against `--color-bg`) — this text color is a Sigma renderer setting bridged from a `--color-text` token (see Theming Bridge below), not raw CSS. |
| Label visibility threshold | Sigma's default zoom-based label-hiding is accepted as-is (D-10) — at the full-tree zoomed-out view, most labels may hide to avoid a hairball of overlapping text; labels reveal as the user zooms in. This is expected behavior, not a bug, per D-10. |
| Selected/hover state | Out of scope for Phase 2 (no click interaction is required by TREE-01…04; NAV-04's highlight-on-click is Phase 3). If Sigma's default hover cursor/reducer highlighting fires for free, that's acceptable incidental behavior — do not build dedicated hover/selection visuals this phase. |

## Edge Visual Spec (TREE-01, D-07)

Hundreds of prerequisite edges must not become a hairball:

| Property | Spec |
|----------|------|
| Type | Straight or slightly curved lines between prerequisite → dependent node pairs (Sigma default edge program; curved only if straight edges cause excessive visual overlap at high-fan-in nodes — tune during implementation, start with straight for simplicity per D-07's "no AND/OR visual distinction" minimalism) |
| Color | Neutral, desaturated gray — `--color-edge: #CBD5E1` — deliberately NOT an area color, so edges recede visually behind nodes/rings and don't compete with the area-color legend encoding |
| Opacity | 0.5 (bridged from token alpha, not hardcoded per-edge) — edges must read as structure, not as the dominant visual weight of the tree; at 678 nodes with potentially 1000+ edges, full-opacity edges would create visual noise that defeats "scannable at a glance" (TREE-02) |
| Width | 1px in graph coordinate space, no per-edge width variation (no weighting/highlighting logic exists yet — that's Phase 3's prerequisite-chain highlight, NAV-04) |
| Z-order | Edges render behind nodes (Sigma default) so node icons/rings stay legible where edges converge |
| Arrowheads | None in Phase 2 — direction is already implied by left-to-right tier flow (D-06); arrowheads on 1000+ edges would add clutter without adding information the column-position already provides. Revisit only if user testing in Phase 3 shows direction ambiguity. |

## Loading State (D-08)

One-shot elkjs layout runs at app load (async, potentially real wall-clock time at 678 nodes — must benchmark, per D-08, but the UI must show *something* regardless of duration):

| Property | Spec |
|----------|------|
| Trigger | Shown from the moment `tech.json` fetch begins until Sigma has a computed layout and first paints |
| Visual | Centered overlay card on `--color-bg`, `--color-surface` background, `--space-lg` padding, subtle shadow, containing: a simple indeterminate spinner (CSS animation, no external library — a single rotating border-arc div is sufficient) + text below it |
| Copy | Heading: "Loading the tech tree" · Body: "Laying out 678 technologies — this takes a few seconds." (Body copy references the real count from `meta.techCount` if easily available at that point in the load sequence; static "678" is an acceptable Phase 2 default if wiring the live count in before layout completes adds complexity) |
| Behavior | Fully blocks interaction with the canvas region (canvas is empty/not-yet-rendered anyway) but the header bar still renders immediately (app title visible right away, so the app doesn't look broken/blank during the fetch+layout window) |

## Empty / Error States

| State | Trigger | Copy | Visual |
|-------|---------|------|--------|
| Data fetch error | `tech.json` fetch fails (network error, 404, malformed JSON, schema validation failure) | Heading: "Couldn't load the tech tree" · Body: "The technology data failed to load. Check your connection and try again." · Action: "Retry" button | Centered overlay card, same shell as loading state, `--color-danger` accent on heading text and a thin `--color-danger` left border accent on the card (not a full red card — keep it calm, this is a data tool not a destructive-action confirmation). "Retry" button uses `--color-accent` background, re-triggers the fetch+layout sequence on click. |
| Empty dataset | `tech.json` loads successfully but `techs` is an empty object (should not happen in practice per DATA-01's 678-tech guarantee, but the UI must not blank-screen if it does) | Heading: "No technologies found" · Body: "The loaded data snapshot doesn't contain any technologies. This shouldn't happen — please report it." | Same centered overlay card pattern as error state, neutral (non-red) styling since this isn't a user-actionable error, no retry button (retrying won't fix a genuinely empty snapshot) |

No destructive actions exist in Phase 2 (no delete/remove/reset UI) — the Copywriting Contract's destructive-confirmation row is intentionally "none" for this phase.

## Theming Bridge (D-12)

`app/src/styles/tokens.css` is the single source of truth. At Sigma initialization (and on any future theme swap — v2 dark mode, out of scope now but must not require rework):

1. Read computed values via `getComputedStyle(document.documentElement)` for: `--color-bg`, `--color-edge`, `--color-text` (a new token, near-black `#111827`, used for node labels — add this to tokens.css alongside the others already named in D-11), `--area-physics`, `--area-society`, `--area-engineering`.
2. Pass these resolved values into Sigma's renderer settings (`defaultNodeColor` fallback, `defaultEdgeColor`, label color settings) and into the per-node/per-edge color assignment logic (area ring color keyed by each tech's `area` field) at graph-construction time.
3. No component ever hardcodes a hex value directly — all colors, including the ones bridged into Sigma, originate from `tokens.css`. This is what keeps v2 dark mode a pure token swap across both the DOM and canvas boundary (D-12's explicit requirement).
